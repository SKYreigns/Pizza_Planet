## 1. Sprint Goal

**Objective:** Initialize the Pizza Planet codebase and establish a robust, production-grade local development environment. 

By the end of this sprint, the repository will contain a compiling Next.js 15 application fully integrated with Tailwind CSS, `shadcn/ui`, Framer Motion, and Zustand. Additionally, the local Supabase instance will be running, and the repository will be correctly structured to match the approved `FrontendArchitecture.md` standards.

---

## 2. Tasks

- **Task 0:** Verify Prerequisites (Node.js >= 20, npm/pnpm, Docker Desktop running).
- **Task 1:** Initialize the Next.js 15 App Router project with TypeScript.
- **Task 2:** Install and configure Tailwind CSS v3/v4.
- **Task 3:** Configure global fonts (Plus Jakarta Sans and Manrope) via `next/font`.
- **Task 4:** Install all required third-party dependencies (Zustand, Framer Motion, Supabase SSR, Zod).
- **Task 5:** Initialize `shadcn/ui` and configure the theme using the "Premium Artisanal Pizza System" design tokens (Deep Pizza Orange `#B1241A`).
- **Task 6:** Create the definitive folder structure (`features/`, `components/`, etc.).
- **Task 7:** Initialize the Supabase CLI and start the local Dockerized instance.
- **Task 8:** Configure `.env.local` and `.env.example`.
- **Task 9:** Configure strict ESLint and Prettier formatting rules.
- **Task 10:** Configure Husky and lint-staged for pre-commit hooks.

---

## 3. Task Order

Execution must happen sequentially to prevent configuration conflicts:

0. Verify Node.js and Docker versions.
1. `npx create-next-app@latest` (Initialize base)
2. Dependency installation (`npm install ...`)
3. `npx shadcn-ui@latest init` (Initializes components folder & tailwind config)
4. Font and global CSS configuration (CSS tokens)
5. Folder scaffolding (`mkdir -p app/(customer) features components/shared...`)
6. `npx supabase init` && `npx supabase start`
7. Environment variable setup.
8. Linting & Formatting config (`.eslintrc.json`, `.prettierrc`).
9. Tooling setup (Husky, lint-staged, VSCode settings).

## 9. Common Risks

- **React 19 / Next 15 Peer Dependencies:** `shadcn/ui` or certain animation libraries might throw peer dependency warnings due to React 19 rc. Use `--legacy-peer-deps` strictly if required, but document it.
- **Docker Requirement:** `supabase start` will fail if Docker Desktop is not running or allocated enough memory. Ensure Docker is fully booted before initializing the backend.
- **Path Aliases:** Ensure `tsconfig.json` correctly maps `@/*` to `./*` so `shadcn/ui` imports resolve cleanly.

## 11. Repository Standards

- **Git Strategy:** GitHub repository with branch protection rules. `main` is the production branch and cannot be pushed to directly.
- **Branch Naming Conventions:**
  - `feature/[ticket-id]-[short-desc]` (e.g., `feature/cart-drawer`)
  - `fix/[ticket-id]-[short-desc]` (e.g., `fix/auth-redirect`)
  - `hotfix/[short-desc]` (e.g., `hotfix/stripe-webhook`)
- **Commit Message Conventions:** Conventional Commits (`feat:`, `fix:`, `chore:`, `refactor:`).
- **Pull Request Conventions:** PRs must include a description of changes, a screenshot (if UI changed), and pass all CI checks before merging.
- **Merge Strategy:** "Squash and Merge" is enforced to keep the `main` history linear and clean.

## 13. Testing Foundation

- **Unit Testing Strategy:** Test isolated utility functions and Zustand reducers.
- **Integration Testing Strategy:** Test Server Actions and Supabase queries against the local DB instance.
- **Component Testing Strategy:** Test interactive components (e.g., Cart Drawer, Checkout Form) in isolation.
- **Recommendations:**

## 14. Deployment Preparation

- **GitHub Repository Setup:** Create the repo, invite team members, and configure branch protection on `main`.
- **Vercel Project Creation:** Link the GitHub repo to Vercel. Ensure the framework preset is "Next.js".
- **Preview Deployments:** Enable Vercel Preview deployments for every Pull Request to facilitate stakeholder review.
- **Environment Management:** Use Vercel Environment Variables UI to manage Production and Preview secrets separately from `.env.local`.
- **Success Criteria:** A dummy commit pushed to `main` successfully triggers a Vercel build and deploys to a `.vercel.app` domain without errors.

---

## 15. Security Baseline

- **Secret Management:** Never commit `.env.local`. Use Vercel for cloud secrets and a secure password manager for sharing local keys.
- **Environment Variable Handling:** Next.js `NEXT_PUBLIC_` prefix is strictly audited to ensure no sensitive backend keys (e.g., Supabase Service Role Key) are leaked to the browser.
- **Dependency Auditing:** Run `npm audit` during CI checks.
- **Access Control Principles:** The `owner` role in Supabase must be manually assigned via the database for the first user; no API route can self-escalate to `owner`.
- **Repository Protection:** Require at least 1 approving review on PRs and enforce that CI (Linting/Typecheck) passes before merging.

---

## 16. Monitoring Foundation

- **Error Monitoring:** Plan to integrate **Sentry** to capture frontend React crashes and Next.js backend Server Action errors.
- **Logging Strategy:** Use standardized console methods locally, which will pipe into Vercel Log Drains in production.
- **Analytics Readiness:** Plan to integrate **PostHog** for tracking user behavior (e.g., checkout funnel drop-offs) securely.
- **Deployment Monitoring:** Monitor Vercel build times and Core Web Vitals via the Vercel Analytics dashboard.
